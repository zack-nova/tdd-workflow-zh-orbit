# TDD Development

把测试当作下一个行为的可执行描述。按小步 Red-Green-Refactor 循环工作，优先沿用仓库已有的测试框架、命名、布局、辅助函数和命令。

## 工作流

1. 先定位。
    - 阅读用户请求、相关代码和已有测试。
    - 找到最小的可观察行为或 bug 复现。
    - 先从项目约定里发现目标测试命令，不要凭空发明。

2. RED：写一个失败测试。
    - 为下一个行为添加或更新最窄但有意义的测试。
    - 用期望行为命名测试，不用实现细节命名。
    - 优先测试真实行为；只在慢、脆弱、破坏性或外部边界处使用 mock。
    - 运行目标测试，确认它因为预期原因失败。

3. 验证 RED 状态。
    - 确认新增或修改的测试确实执行了。
    - 语法错误、测试环境损坏、依赖缺失或无关失败都不是有效 RED；先修正这些问题。
    - 只有当编译失败正好表达缺少新行为或新 API 时，才算有效信号。

4. GREEN：实现最少代码。
    - 只有 RED 有效后才改生产代码。
    - 只写足够通过失败测试的实现。
    - 避免无关重构、大范围重写或猜测性选项。
    - 运行同一个目标测试，确认通过。

5. 只在绿色状态下 REFACTOR。
    - 测试通过后，再清理重复、命名、结构或性能。
    - 保持行为不变，并重新运行相关测试。

6. 对下一个行为重复循环。
    - 边界条件和错误路径也用小循环补上。
    - 风险越大，验证范围越要扩大。

## 选择测试

- 纯函数、校验器、解析器、计算、状态流转和数据转换，用单元测试。
- 路由、数据库操作、服务、队列和跨模块行为，用集成或 API 测试。
- 关键用户路径、表单行为、导航、键盘流程和浏览器可见结果，用 UI 或 E2E 测试。
- 无预期行为变化的重构，先运行已有覆盖；风险较高时先补特征测试。
- 已有实现只要求补覆盖时，说明可能没有 RED；但断言要足够强，能抓住真实回归。

## 测试质量

- 断言可观察行为和公共契约，不断言私有实现细节。
- 测试保持独立、确定、具体。
- 覆盖 happy path、边界值、非法输入、空输入和有意义的错误路径。
- UI 测试避免硬编码 sleep；优先使用语义定位和可观察状态。
- 不接受证明力太弱的“通过测试”。

## 验证

先运行最小有用命令，再按风险扩大范围

结束前报告：

- RED 证据，包括失败测试和失败原因为什么符合预期；
- GREEN 证据，包括通过的命令；
- 已运行、跳过或不可用的更大范围检查。
