# Review Commit Debt

## 概览

用这个流程把工作区变成经过 review、必要修复、验证、提交，并明确留下后续技术债的代码。流程保持轻量、repo-local、以证据为准。

默认执行 `review + 提交 + 记录技术债`。只有用户明确限定时，才降级为只 review、只提交或只记录技术债。

## 工作流

### 1. 确认范围

- 阅读用户请求，判断用户是否明确限定为 `只 review`、`只提交`、`review + 提交`，或其它更窄范围；没有明确限定时，按 `review + 提交 + 记录技术债` 处理。
- 编辑或提交前先检查仓库状态：
    - `git status --short`
    - `git branch --show-current`
    - `git diff --stat`
    - `git diff`
    - 如果已有暂存内容，运行 `git diff --cached`
- 把已有或无关改动视为用户所有。不要 revert。除非用户明确要求，否则不要把它们放进提交。
- 如果多个逻辑改动混在一起，优先拆成多个原子提交。只有边界有风险或无法从 diff 推断时才询问。

### 2. Review 代码

先看行为，再看可维护性：

- 确认实现是实质性的：文件存在不够。检查代码是否接入调用点、使用真实数据路径，并具备功能行为。
- 查找占位模式，例如 `TODO`、`FIXME`、`XXX`、`HACK`、`PLACEHOLDER`、"coming soon"、空 handler、`return null`、`return undefined`、`return {}`、`return []`、只打日志的 handler、硬编码 demo 数据，以及遗留在用户可见输出里的模板占位文本。
- 对照用户任务检查测试、实现和用户可见行为。按相关性标记缺失的边界情况、错误处理、校验、鉴权/权限、持久化、并发、迁移、启动行为和回滚路径。
- 运行便宜且本地的静态检查，例如 `git diff --check`、lint、类型检查或格式检查，前提是仓库提供这些命令。
- 如果是只 review 请求，先列发现的问题，按严重程度排序，并包含文件/行号。没有发现问题时明确说明，并提到剩余风险或未运行测试。

### 3. 修复或报告

- 对 review 后能安全修复的问题，直接修复。修复必须在用户请求范围内、改动边界清楚，并能用本地检查或测试验证。
- 修复时保持小步，不做无关重构。必要时使用 `$tdd-development` 的 Red-Green-Refactor 节奏补测试或回归用例。
- 如果发现重大问题，例如安全风险、数据丢失风险、权限绕过、迁移/持久化风险、破坏公共契约、验证持续失败，或修复范围已经超出请求，不要继续做普通 ready 提交；先向用户报告问题、证据、风险和建议路径。
- 能修复的重大问题可以先修复，但仍要在最终回复中报告问题性质、修复方式和剩余风险。

### 4. 提交前验证

- 优先运行覆盖已改行为的聚焦测试；影响面大时再运行更大范围测试。
- 先从项目约定发现命令，不要凭空发明：检查 `package.json`、`pyproject.toml`、`Makefile`、CI 配置或项目文档。
- 如果本轮改过代码，保留相关测试证据：
    - 运行的命令
    - 通过/失败结果
    - 重要失败原因
- 如果因为依赖、服务、凭证或时间无法运行测试，说明阻塞原因，并做最强可用的静态/手工验证。
- 相关验证失败时，不要创建普通“ready”提交，除非用户明确接受风险。

### 5. 安全提交

- 暂存前立刻重新检查 `git status --short`。
- 只暂存属于目标逻辑改动的文件。有无关改动时使用路径限定暂存。
- 用 `git diff --cached --stat` 和 `git diff --cached` 检查暂存内容。
- 使用小而原子的提交。独立改动拆成多个提交。
- 合适时优先使用 Conventional Commit 风格：
    - `fix: correct login redirect`
    - `feat: add invoice export`
    - `test: cover retry failure`
    - `docs: record queue cleanup debt`
    - `refactor: simplify parser state`
- 有帮助时写简短正文：改了什么、为什么改、验证证据。
- 提交后再次运行 `git status --short`，报告仍未提交的文件。

### 6. 记录技术债

只记录 review/提交后仍未解决的债务。不要记录模糊愿望或已经修复的问题。

- review 时发现的技术债记录到仓库专门的技术债 issue，不写入 repo 内技术债文件。
- 如果专门 issue 不存在，按仓库、团队、用户或当前项目 issue 管理要求创建技术债 issue；如果要求不明确、权限不可用或目标不明确，先询问。
- 条目保持简短、可执行：

```markdown
- Debt: <具体未解决问题>
- Impact: <风险、维护成本或用户影响>
- Evidence: <文件、测试、日志或 review 观察>
- Next step: <最小可用后续动作>
```

- 如果暂时无法访问 issue，把要发布的 issue 评论草稿写到 `.development-audit/`，并在最终回复里说明还需要发布。
- 如果债务严重到阻塞发布，不要只埋进 issue。最终回复中也要明确指出，并避免“ready”表述。
- 如果没有遗留债务，直接说明，不要添加空的 issue 评论。

## 最终回复

保持简洁：

- 说明 review 和提交了什么。
- 如果创建了提交，包含 commit hash。
- 列出验证命令和结果。
- 总结记录的技术债，以及更新的技术债 issue。
- 提到提交范围外仍未提交的改动。
